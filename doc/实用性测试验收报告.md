# 实用性测试验收报告
## 1. 测试概述
验证系统在日常使用场景下的可用性与效率（Usability / Practicality）。重点检验管理员与患者常见操作流程的便捷性、错误提示的友好程度、表单/输入校验体验以及系统在典型并发（并发挂号）下的行为表现。

## 2. 测试流程与执行记录
### 2.1 测试用例示例
| 用例ID | 场景 | 操作步骤 | 预期 | 实际 | 状态 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| USR-001 | 管理员登录并查看医生列表 | 登录→导航到 管理→医生 列表 | 页面加载 < 2s，列表内容完整 | 页面加载 ~1s，内容完整 | 通过 |
| USR-002 | 管理员新增医生并设置排班 | 新增医生→填写必填→保存 | 提示成功并可见新医生 | 保存成功 | 通过 |
| USR-003 | 患者使用手机号注册并预约 | 注册→登录→选择医生→提交预约 | 提示预约成功或候补信息 | 提示明确（通过/候补） | 通过 |
| USR-004 | 表单校验友好性 | 提交缺失必填字段 | 前端阻止并高亮字段，返回错误提示 | 部分字段未高亮（体验欠佳） | 不通过 |
| USR-005 | 并发挂号 (50 并发) | 多用户同时提交挂号请求 | 系统正确分配号源或候补，无崩溃 | 部分请求延迟上升，未崩溃 | 通过（性能需观测） |

### 2.2 关键观测
- 表单校验在大多数关键场景覆盖良好，但存在少数字段未触发高亮提示；
- 并发场景下系统能保持可用，但响应时间在高并发下上升明显（建议做性能优化）；
- 错误信息多为可读信息，但部分接口错误返回含技术栈堆栈或异常文字，需统一错误封装。

## 3. 结果评估
- 结论：总体可用，但存在 UX 小问题与性能可优化点（表单提示不一致、并发响应时间需要改善）。

## 4. 通过时的下一版本改进意见
1. 统一前端表单校验组件与错误高亮策略，确保所有必填与校验错误均有视觉提示；
2. 在高频路径中加入轻量的客户端节流或 debounce，减少短时并发压力；
3. 后端在并发关键路径（挂号）引入更细粒度锁或乐观并发控制，并增加性能监控与指标告警。

## 5. 不通过时的整改意见
1. 表单校验体验不一致：修复前端校验逻辑并补充 E2E 用例覆盖（责任：前端 @前端）；期限：3 个工作日；
2. 性能问题（高并发延迟）：进行热点分析（DB 查询、事务、锁），增加缓存或改写临界路径，责任：后端 + DBA（@后端/@DBA）；期限：5 个工作日；
3. 错误信息泄露实现不一致：统一错误返回格式并屏蔽敏感信息，责任：后端（@后端）；期限：2 个工作日。

## 6. 责任划分（建议）
- 前端：修复表单与错误展示、增加 E2E 覆盖；
- 后端：优化并发挂号路径、统一错误封装；
- 测试：补充 E2E 用例并在 CI 上跑自动化可用性验证；
- DBA/运维：监控与调优数据库性能。
